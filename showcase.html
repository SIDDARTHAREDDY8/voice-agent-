Every call below is generated live and scored against a known ground truth.

Verification layer

✓
PASS — all required fields captured & internally consistent The same check fires ⚠ if deductible-met > deductible.
  • Active coverage, plan type & effective date confirmed
  • Deductible-met ≤ deductible (the trap that was caught)
  • Out-of-pocket met ≤ out-of-pocket max
  • Call reference number captured for appeals
  • Result mapped to FHIR CoverageEligibilityResponse

Which calls need a human?

python review_queue.py

You can't review millions of calls a year — and you can't blindly auto-post data that might be wrong. So every finished call is scored per-field and routed: auto-post when it's confident and consistent, human review when it isn't. Humans only touch the fraction the system is unsure about.

3 of 4 calls auto-post — zero human attention. A team reviews only the 25% the system flags. That's the lever that lets a handful of people stand behind millions of calls without scaling headcount linearly.

Honestly scoped: voice is modeled as text turns — the same loop drops onto a telephony stack (Twilio / LiveKit) with STT+TTS; call.py is the seam. The payer is simulated so accuracy is measurable, and the verification rules are hand-written today — the next step is learning them from labeled call outcomes. I scoped this to the reasoning + reliability layer on purpose: that's the hard part.
